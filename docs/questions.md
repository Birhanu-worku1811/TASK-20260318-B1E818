### 1. Promotion Stacking and Hierarchy
**Question:** The prompt lists multiple promotion types (spend-and-save, tiered pricing, etc.), but does not define how they interact if a single item qualifies for multiple rules.  
**Assumption:** Promotion rules are applied in a strict hierarchy (Item-level > Order-level > Global) rather than being fully additive.  
**Solution:** Implemented a `PromotionEngine` that calculates the "Best Deal" for the customer by simulating different rule combinations or applying them in a defined priority sequence.

### 2. Encryption Key Management
**Question:** For "field-level encryption" in a single-machine offline Docker environment, it is unclear where the master encryption key is securely stored.  
**Assumption:** The master key is not stored in the database but is provided as a Docker environment variable or a protected file on the host machine.  
**Solution:** Used a dedicated configuration module that loads a `MASTER_ENCRYPTION_KEY` from the environment, ensuring sensitive fields are unreadable if the database file is stolen.

### 3. Printing Hardware Interface
**Question:** The system "abstracts receipt printing," but communicating from a Docker container to a physical USB or Serial printer requires specific driver mapping.  
**Assumption:** Printers are either network-based (TCP/IP) or mapped to the Docker container as a character device (e.g., `/dev/usb/lp0`).  
**Solution:** Created a printer driver service that supports ESC/POS commands via network sockets as the default, with a fallback for local device path mapping.

### 4. Initial Admin Provisioning
**Question:** The prompt requires usernames/passwords but does not specify how the first "Operation Administrator" is created in a fresh offline install.  
**Assumption:** A default "admin" account is created via a database seed script during the initial Docker container setup.  
**Solution:** Developed a CLI "seed" command that initializes the RBAC (Role-Based Access Control) tables and creates an initial admin user with a temporary password.

### 5. Feature Library Calculation Trigger
**Question:** "Sliding window statistics" can be resource-heavy; it is unclear if these are calculated in real-time during a transaction or asynchronously.  
**Assumption:** Real-time calculations are limited to critical checkout logic, while broader "correlation calculations" are handled by background tasks.  
**Solution:** Utilized FastAPI's `BackgroundTasks` to update the Feature Library asynchronously, preventing the API response time from slowing down during high-traffic checkout moments.

### 6. Notification Delivery Scope
**Question:** For an "offline" system, it is unclear if "Notification Center" messages stay purely within the web UI or need to reach external devices.  
**Assumption:** Notifications are strictly internal (in-site) and delivered via WebSockets to active users or stored in an "Inbox" for later viewing.  
**Solution:** Implemented a WebSocket manager to push "Pending Approval" or "Budget Alert" pop-ups to logged-in users in real-time without requiring a page refresh.

### 7. Audit Log Protection
**Question:** The prompt requires an "immutable audit table," but standard SQL tables can be edited by anyone with database access.  
**Assumption:** Immutability is enforced at the database level via triggers to prevent `UPDATE` or `DELETE` operations on that specific table.  
**Solution:** Added a PostgreSQL trigger function to the `audit_logs` table that raises an exception if any modification or deletion is attempted on existing records.

### 8. Barcode Scanner Input Handling
**Question:** It is unclear if "quick matching" for barcodes assumes a standard HID (Keyboard-emulated) scanner or an API-driven hardware integration.  
**Assumption:** The frontend/API assumes standard string input, allowing HID scanners to "type" the barcode directly into the search field.  
**Solution:** Designed the retrieval API to be idempotent and high-performance (sub-50ms) to handle the rapid "enter" keys generated by hardware scanners.

### 9. File Integrity Verification Timing
**Question:** File fingerprints are required, but the prompt doesn't specify when the verification should occur (at upload, at download, or periodically).  
**Assumption:** Integrity is verified both during the initial upload and every time a project reviewer attempts to open/download the attachment.  
**Solution:** Stored a SHA-256 hash in the `Attachment` metadata; the system re-calculates the file's current hash and compares it against the stored value before every file read operation.

### 10. Return/Exchange Payment Logic
**Question:** For "reverse settlements," it is unclear how the system handles returning money to multiple payment methods (e.g., partial cash, partial card).  
**Assumption:** Refunds are returned to the original payment methods in the same proportion as the original transaction.  
**Solution:** Implemented a "Linked Refund" model that references the original `PaymentRecord` and ensures the total refunded via any single method does not exceed the amount originally paid through that method.
